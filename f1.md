F1: 核心功能单元一
主要功能是维护项目元数据，将Repo的信息清晰地展示在META.md中，方便用户和AI理解和使用。

元数据的组织原则是：
0. 元信息体现且只体现项目当前的状态
1. 元信息的目的就是把这个事情描述清楚，把一个事情描述清楚的方式是先直观，再宏观，再微观
2. 直观层面，要让人直观的知道这是一个什么事，要包括一段对于项目整体的总结性描述
3. 宏观层面，要让人过看之后有一个完整的信息结构
4. 微观层面，不存储细节信息，要做到知道信息去哪儿找，是以什么形式存在的，如何获取
5. 所有信息必须标明来源，引用或者总结了Repo的什么内容。
6. 不得有任何编造和推断的信息。

元数据的存储结构， 项目目录下的：
.laddermoon/
meta.md
meta-commit-id

Question数据的定义：
在.laddermoon/questions/目录下，
question-1.md
question-2.md
question-meta.jsonl # {"id": 1, "status": "open"} #status: open, solved, issued, skipped
last-question-id # 1 

Issue数据的定义：
在.laddermoon/issues/目录下，
issue-1.md
issue-2.md
issue-meta.jsonl # {"id": 1, "status": "open"} #status: open, solved, skipped
last-issue-id # 1


核心是四个Skill
1. laddermoon-f1-inspector 
2. laddermoon-f1-clarifier
3. laddermoon-f1-synchronizer #将Repo中的变化同步到元数据
4. laddermoon-f1-initializer #初始化元数据

laddermoon-f1-inspector #检查元数据的清晰度
输入： question文件名
1. 读取元数据
2. 根据下面给出的提问逻辑提出问题
3. 将问题写入Question文件
提问的逻辑是：
1. 元数据的组织原则是否满足
2. 从一个想要全面了解项目的项目管理者的视角来看，元数据是否足够清晰，是否有某些关注的信息有缺失


laddermoon-f1-clarifier #回答Inspector的Question，如果回答不了，向Repo提交Issue
输入：  question文件名， issue文件名
1. 读取Question文件
2. 根据Question文件，去Repo中找答案
3. 如果找到，完善元数据。
4. 创建Issue文件


laddermoon-f1-synchronizer #将Repo中的变化同步到元数据
1. 读取meta-commit-id文件，获取元数据已经更新到的Commit ID
2. 读取Repo的最新Commit ID
3. 如果有变化，则读取Repo中从meta-commit-id到最新Commit ID之间的变化，如果有需要，更新meta.md
4. 更新meta-commit-id文件


laddermoon-f1-initializer #初始化元数据
1. 根据元数据的要求，逐步从Repo中提取信息，构造meta.md
2. 更新meta-commit-id文件



元数据维护的流程：
1. laddermoon-f1-initializer #初始化元数据 ， 只做一次

# 每次使用元数据之前
2. laddermoon-f1-synchronizer #先与Repo同步
3. laddermoon-f1-inspector #然后检查元数据是否足够清晰，如果不够清晰，创建Question文件
4. 检查是否有question文件创建。
5. laddermoon-f1-clarifier #回答Question文件，如果回答不了，向Repo提交Issue
不断重复345，直到3不创建Question文件。
如果有Issue，说明项目本身中存在结构不完整或者信息缺失。需要通过开发工作来补全。

目标是通过上面的流程，让元数据足够清晰，方便用户和AI理解和使用。
