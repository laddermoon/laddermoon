# LadderMoon 产品文档

## 产品定位

**LadderMoon** 是一个 AI 驱动的项目管理工具，核心理念是 **"AI AS ME"** —— 让 AI 成为开发者决策逻辑的"影子自我"，通过学习用户的架构偏好和决策模式，自动化地管理和推进项目。

### 品牌含义

- **Ladder (梯子)**：代表通过 META 元信息建立的层级结构，步步稳健向上攀爬
- **Moon (月亮)**：代表项目的最终愿景和目标
- **整体寓意**：通过严密的元架构（梯子），触达原本遥不可及的完美代码（月亮）

---

## 核心概念

### META 元信息库

项目的"大脑"，一个独立于项目代码的信息库，用于存储项目的元信息。

**特点：**
- 独立于项目本身，项目没有此信息库也能完整运行
- 不包含 API 定义、产品原型等项目本身应有的信息
- 记录元信息的描述（如：产品原型文档在哪、格式是什么、如何查看）
- 采用 **影子分支 (Shadow Branch)** 方式管理，如 `ai-brain` 分支

**组织原则：**
- 单一文件，方便 AI 阅读
- 重要信息在上，次要信息在下
- 下层内容需与上层内容对齐

### 三维度驱动机制

AI 通过三个维度推动项目演进：

| 维度 | 说明 | 处理方式 |
|------|------|----------|
| **疑问 (Question)** | 对项目现状不清楚的地方 | AI 可通过代码库自行解决，或由用户解答 |
| **问题 (Issue)** | 项目现状中存在的缺陷 | 用户判断是否为真问题及优先级 |
| **改进 (Suggestion)** | 让项目变得更好的建议 | 用户判断方向是否正确 |

## 命令行接口

CLI 工具名：`lm`（LadderMoon 缩写）

### 核心命令

| 命令 | 功能 |
|------|------|
| `lm init` | 初始化影子分支和 META 结构 |
| `lm feed [user input text]` | 录入用户提供的信息 |
| `lm sync` | 将代码库变化同步到元信息库 |
| `lm audit` | 启动 Issuer 角色，探测潜在问题, 每个Issue都记录到一个文件中 |
| `lm propose` | 启动 Suggester 角色，提出改进建议，每个proposal都记录到一个文件中 |
| `lm solve [filename]` | 针对特定 Issue/Suggestion 启动 Coder |
| `lm status` | 查看 META 状态、CommitID 及待处理清单 |

## 用户交互设计

### 交互原则

- 终端显示不使用 "AI is thinking..."
- 使用个性化提示，如：
  - `[LadderMoon] Synchronizing your intent...`
  - `[LadderMoon] Reflecting your architectural preferences...`

### 用户介入点

1. **疑问解答**：当 AI 无法从代码库解决时
2. **问题分类**：判断 Issue 是否为真问题，设定优先级
3. **建议评估**：判断 Suggestion 方向是否正确
4. **代码审核**：Reviewer 环节的最终确认

---

## 决策日志

记录所有问题和改进建议的处理结果，用于：
- 追溯决策历史
- 为 AI 自我提升提供训练数据
- 优化 Issuer 和 Suggester 的提问能力

**记录内容：**
- 用户对问题的判断（是否为真问题、优先级）
- 用户对建议的反馈（方向是否正确、替代方案）
- 延迟决策的事项

---

## 目标用户

独立开发者和小型团队，希望借助 AI 能力提升项目管理效率，同时保持对项目的完全控制。
